## Abstract

This EIP defines a **state-augmentation standard** built on [ERC-721](./eip-721.md). The EVM state model is inherently stateless with respect to time: it records what *is*, not what *was*. Smart contracts cannot read event logs. All prior ownership evidence is therefore off-chain by construction — fragile, trust-dependent, and non-composable.

ERC-8169 extends the EVM state model to include a **persistent, append-only provenance layer** that other contracts can read natively. This is not an NFT-tracking upgrade. It is a new state category: ownership data with a temporal dimension, encoded in storage slots rather than event logs, and composable across any on-chain logic.

The standard introduces a **three-layer ownership model**:

| Layer | Name | Mutability | Purpose |
|:------|:-----|:-----------|:--------|
| 1 | Immutable Origin | Write-once | Records who minted / created the token |
| 2 | Historical Trail | Append-only | Chronological list of every past owner |
| 3 | Current Authority | Mutable | Standard ERC-721 `ownerOf()` semantics |

A compliant contract MUST implement [ERC-721](./eip-721.md), [ERC-165](./eip-165.md), and the `IERC721H` interface defined herein.

## Motivation

### Positioning: What This Standard Is

ERC-8169 is a **state-augmentation standard**. Its purpose is not to improve NFT tracking tooling or replace indexers for display purposes. Its purpose is to extend the **EVM state model** — the set of things other smart contracts can natively read and compose against — to include a temporal ownership dimension.

This distinction matters for standardization: a standard that helps frontends show richer data is a product feature. A standard that lets a DAO contract grant privileges based on verifiable on-chain provenance without requiring trusted off-chain infrastructure is a primitive — one that changes what is possible to build, not just how it is displayed.

**This ERC is the latter.** Positioning it correctly is necessary for evaluation by the community.

### L2-Native Design Rationale

The write-overhead of maintaining a three-layer provenance state is intentional and explicitly sized for L2 deployment. On Ethereum L1, the storage write cost for FULL-mode history makes this standard economically impractical for all but high-value provenance use cases. On L2 rollups (Arbitrum, Base, Optimism, Starknet), write costs are sufficiently low that provenance storage becomes viable at scale.

This is not a compromise. This standard is designed as an **L2-native provenance primitive** — a foundational building block for a class of on-chain applications that require composable, trustless custody chains, and that can only exist at the cost structure L2 enables.

Deployments on L1 are valid and supported. They represent an explicit trade-off: provenance permanence over gas efficiency, appropriate for high-value assets where provenance is the economic core of the token's value.

### The EVM State Gap

ERC-721 stores exactly one piece of ownership data: the **current** owner. When Alice transfers Token #1 to Bob, and Bob transfers to Charlie, there is no storage-level ownership record accessible to other smart contracts proving Alice ever held the token. The only evidence lives in `Transfer` event logs, which:

1. **Cannot be read by other smart contracts** — `eth_getLogs` is an off-chain JSON-RPC call, not an EVM opcode.
2. **Require off-chain indexers** — The Graph, Alchemy, or custom infrastructure must be trusted to reconstruct history.
3. **Are fragile** — Indexers go down, have lag, and cost money. Archive nodes are expensive.
4. **Cannot power on-chain governance** — A DAO cannot grant "founding member" privileges based on event logs alone.

This proposal does not aim to replace event logs. It aims to fill a structural gap in the EVM state model: **ownership data that other smart contracts can read and act on natively, without off-chain intermediaries**.

### Composable Provenance: What the Primitive Enables

The following are examples of on-chain logic that requires provenance as a state-accessible primitive. Each is either impossible or trust-dependent without this standard:

1. **Art Provenance**: A collector wants on-chain proof that a Picasso NFT passed through Christie's gallery. ERC-721 cannot answer "did address X ever own token Y?" without an indexer.

2. **Founder Benefits**: A project wants to give perpetual 2% royalties to the original minter of each token, even after they sell. ERC-721 discards the minter address on first transfer.

3. **Early Adopter Airdrops**: A protocol wants to airdrop governance tokens to everyone who held an NFT during the first 100 blocks. With ERC-721, this requires reconstructing history from logs off-chain and submitting a Merkle root — expensive, slow, and trust-dependent.

4. **Proof-of-Custody**: In a dispute over rightful ownership, on-chain storage provides a directly queryable chain of custody. Event logs require off-chain reconstruction and indexer infrastructure; storage slots are natively readable by any smart contract.

5. **Gaming Veteran Status**: A game wants to show a "Day 1 Player" badge to anyone who minted during launch week, regardless of whether they still hold the token.

### Why State Augmentation, Not a New Token Standard?

This standard extends ERC-721 interfaces and storage semantics without introducing a new token primitive. The decision is architectural: a new token standard would sever compatibility with every existing wallet, marketplace, and library. ERC-8169 augments the state model while preserving ERC-721 integration surfaces.

Implementations MAY introduce behavioral constraints (for example transfer cooldowns or self-transfer prevention) to protect history integrity and mitigate griefing. Therefore this is not always a strict behavioral superset of ERC-721, but a compatible, constrained extension.

The provenance layer is additive. Contracts that do not require historical queries ignore it entirely. Contracts that do can compose against it without any trusted off-chain component. This is the correct place to add a temporal state dimension: not as a competing standard, but as an extension of the existing one at the storage level.

## Specification

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119 and RFC 8174.

### Interface

This standard defines two interfaces:

- **`IERC721H`** (Core) — MUST be implemented. Origin tracking, historical ownership, O(log n) block resolution, and pagination.
- **`IERC721HAnalytics`** (Optional) — MAY be implemented. Convenience queries: provenance reports, early adopter detection, per-address enumeration, and supply tracking.

#### Core Interface (`IERC721H`)

```solidity
// SPDX-License-Identifier: CC0-1.0
pragma solidity >=0.8.0;

interface IERC721H {
    // ── Events ────────────────────────────────────────

    /// @notice Emitted every time a token's ownership history is extended.
    event OwnershipHistoryRecorded(
        uint256 indexed tokenId,
        address indexed newOwner,
        uint256 timestamp
    );

    /// @notice Emitted once at mint to record the immutable creator.
    event OriginalCreatorRecorded(
        uint256 indexed tokenId,
        address indexed creator
    );

    /// @notice Emitted when a token is burned (Layer 3 cleared).
    event HistoricalTokenBurned(uint256 indexed tokenId);

    // ── Layer 1: Immutable Origin ─────────────────────

    /// @notice Returns the address that originally minted `tokenId`.
    function originalCreator(uint256 tokenId) external view returns (address);

    /// @notice Returns the block number at which `tokenId` was minted.
    function mintBlock(uint256 tokenId) external view returns (uint256);

    /// @notice Returns true if `account` was the original minter of `tokenId`.
    function isOriginalOwner(uint256 tokenId, address account) external view returns (bool);

    // ── Layer 2: Historical Trail ─────────────────────

    /// @notice Returns true if `account` has ever owned `tokenId`. MUST be O(1).
    function hasEverOwned(uint256 tokenId, address account) external view returns (bool);

    /// @notice Returns the complete chronological ownership chain for `tokenId`.
    function getOwnershipHistory(uint256 tokenId)
        external view returns (address[] memory owners, uint256[] memory timestamps);

    /// @notice Returns the owner of `tokenId` at any arbitrary `blockNumber`.
    /// @dev Uses O(log n) binary search over chronological `_ownershipBlocks`.
    ///      Returns the owner who held the token at `blockNumber`, even if no
    ///      transfer happened at that exact block. Returns address(0) if the
    ///      token was not yet minted at `blockNumber`.
    ///      Uses block.number (not block.timestamp) to prevent validator manipulation.
    function getOwnerAtBlock(uint256 tokenId, uint256 blockNumber) external view returns (address);

    // ── Pagination Helpers (Anti-Griefing) ────────────

    /// @notice Returns the total number of entries in `tokenId`'s ownership history.
    function getHistoryLength(uint256 tokenId) external view returns (uint256);

    /// @notice Returns a paginated slice of ownership history.
    function getHistorySlice(uint256 tokenId, uint256 start, uint256 count)
        external view returns (address[] memory owners, uint256[] memory timestamps);

    // ── Layer 3: Current Authority ────────────────────

    /// @notice Returns true if `account` is the current owner of `tokenId`.
    function isCurrentOwner(uint256 tokenId, address account) external view returns (bool);

    // ── Lifecycle ─────────────────────────────────────

    /// @notice Burns `tokenId`. Layer 1 and Layer 2 data SHOULD be preserved.
    function burn(uint256 tokenId) external;
}
```

#### Analytics Extension (`IERC721HAnalytics`)

Implementations MAY additionally implement the following convenience interface. Support is signaled via [ERC-165](./eip-165.md).

```solidity
// SPDX-License-Identifier: CC0-1.0
pragma solidity >=0.8.0;

/// @title IERC721HAnalytics — Optional analytics extension
interface IERC721HAnalytics {
    /// @notice Returns how many times `tokenId` has been transferred (excludes mint).
    function getTransferCount(uint256 tokenId) external view returns (uint256);

    /// @notice Returns every token `account` has ever owned (deduplicated).
    function getEverOwnedTokens(address account) external view returns (uint256[] memory);

    /// @notice Returns every token `creator` originally minted.
    function getOriginallyCreatedTokens(address creator) external view returns (uint256[] memory);

    /// @notice Returns true if `account` minted any token at or before `blockThreshold`.
    function isEarlyAdopter(address account, uint256 blockThreshold) external view returns (bool);

    /// @notice Returns a full provenance report for `tokenId` in a single call.
    function getProvenanceReport(uint256 tokenId)
        external view returns (
            address creator,
            uint256 creationBlock,
            address currentOwnerAddress,
            uint256 totalTransfers,
            address[] memory allOwners,
            uint256[] memory transferTimestamps
        );

    /// @notice Returns the total number of tokens currently in existence (excludes burned).
    function totalSupply() external view returns (uint256);

    /// @notice Returns the total number of tokens ever minted (includes burned).
    function totalMinted() external view returns (uint256);

    // ── Per-Address Pagination ────────────────────────

    /// @notice Returns the number of distinct tokens `account` has ever owned.
    function getEverOwnedTokensLength(address account) external view returns (uint256);

    /// @notice Returns a paginated slice of tokens `account` has ever owned.
    function getEverOwnedTokensSlice(address account, uint256 start, uint256 count)
        external view returns (uint256[] memory tokenIds);

    /// @notice Returns the number of tokens `creator` originally minted.
    function getCreatedTokensLength(address creator) external view returns (uint256);

    /// @notice Returns a paginated slice of tokens `creator` originally minted.
    function getCreatedTokensSlice(address creator, uint256 start, uint256 count)
        external view returns (uint256[] memory tokenIds);

    /// @notice DEPRECATED: Use getOwnerAtBlock() from IERC721H instead.
    function getOwnerAtTimestamp(uint256 tokenId, uint256 timestamp) external pure returns (address);
}
```

### Behavioral Requirements

#### Layer 1 — Immutable Origin

1. `originalCreator(tokenId)` MUST be set exactly once, at mint time.
2. `originalCreator(tokenId)` MUST NOT change after mint, even if the token is transferred or burned.
3. `mintBlock(tokenId)` MUST record `block.number` at the time of minting.
4. The contract MUST emit `OriginalCreatorRecorded(tokenId, creator)` exactly once per token, at mint.

#### Layer 2 — Historical Trail

5. On every transfer (including mint), the recipient address MUST be appended to the ownership history for `tokenId`.
6. The ownership history MUST be append-only. Entries MUST NOT be removed, modified, or reordered.
7. `hasEverOwned(tokenId, account)` MUST execute in O(1) time. Implementations SHOULD use a nested `mapping(uint256 => mapping(address => bool))`.
8. `getOwnershipHistory(tokenId)` MUST return two parallel arrays of equal length: addresses and timestamps, in chronological order. The first element MUST be the original creator.
9. The `_everOwnedTokens` mapping MUST be deduplicated — if an address receives the same token twice (e.g., Alice → Bob → Alice), the token ID MUST appear only once in Alice's list.
10. The contract MUST emit `OwnershipHistoryRecorded(tokenId, newOwner, timestamp)` on every mint and transfer.

All pagination functions (both core and analytics) MUST return an empty array if `start >= length`. They MUST NOT revert.

#### Layer 3 — Current Authority

11. Layer 3 MUST implement ERC-721 interfaces and authorization semantics for `ownerOf()`, `balanceOf()`, `transferFrom()`, `safeTransferFrom()`, `approve()`, `setApprovalForAll()`, `getApproved()`, and `isApprovedForAll()`. Implementations MAY add explicit anti-griefing constraints (such as self-transfer rejection and transfer cooldown) and MUST document those divergences.

#### Sybil Protection (Dual-Layer)

17. Contracts SHOULD implement intra-transaction protection to block multiple transfers of the same token within one transaction (A→B→C chains). If transient storage ([EIP-1153](./eip-1153.md)) is available on the target chain, implementations SHOULD use it for this purpose. Transient storage keys MUST be namespaced (e.g., `keccak256(tokenId, constant)`) to prevent collision with other contracts or libraries in the same transaction context.
18. Contracts SHOULD implement inter-transaction protection to enforce one owner per token per block number across separate transactions. The reference implementation derives this from `lastTransferBlock[tokenId]` — if the last recorded block equals `block.number`, the transfer reverts. When implemented, `block.number` MUST be used instead of `block.timestamp` to prevent validator manipulation.
19. `getOwnerAtBlock(tokenId, blockNumber)` MUST return the owner of `tokenId` at any arbitrary `blockNumber`, not just blocks where a transfer occurred. Implementations SHOULD use O(log n) binary search over the chronological `_ownershipBlocks` array to resolve the last transfer at or before `blockNumber`. Returns `address(0)` if the token was not yet minted at `blockNumber`. In FLAG_ONLY and COMPRESSED modes, this function MAY return `address(0)` for all queries since ownership arrays are not stored.
20. Contracts MAY implement a configurable transfer cooldown (`transferCooldownBlocks`) that enforces a minimum number of blocks between consecutive transfers of the same token. This mitigates L2 cross-block churn griefing. When set to 0 (default), only the same-block guard applies. Implementations SHOULD document whether the cooldown anchor initializes at mint via `lastTransferBlock`.
21. Contracts MUST prevent self-transfers (`from == to`) to avoid polluting Layer 2 history without actual ownership change.

#### COMPRESSED Mode Verification Model

22. In COMPRESSED mode, the initial commitment MUST be deterministic: `H0 = keccak256(abi.encodePacked(bytes32(0), owner0, block0, timestamp0))`, where `(owner0, block0, timestamp0)` is the mint record.
23. For each subsequent ownership step `i > 0`, implementations MUST update commitment as `Hi = keccak256(abi.encodePacked(H(i-1), owneri, blocki, timestampi))`.
24. Implementations MAY store only the latest commitment (`Hn`) on-chain. When only `Hn` is stored, on-chain inclusion proofs for intermediate steps are NOT available by default.
25. A verifier proves a claimed full sequence by reconstructing the ordered sequence off-chain, replaying the recurrence from `H0`, and comparing computed `Hn` with on-chain `getHistoryHash(tokenId)`.
26. Merkle inclusion proofs are OPTIONAL and require an additional data structure not defined by this standard. COMPRESSED hash-chain mode alone does not provide Merkle membership proofs.

#### Burn Behavior

12. When a token is burned, Layer 3 data (current owner) MUST be cleared.
13. When a token is burned, implementations SHOULD preserve Layer 1 data (`originalCreator`, `mintBlock`). Implementations that do not preserve MUST document this behavior.
14. When a token is burned, implementations SHOULD preserve Layer 2 data (`_ownershipHistory`, `_ownershipTimestamps`, `_hasOwnedToken`). Implementations that do not preserve MUST document this behavior.
15. When Layer 1 and Layer 2 data are preserved after burn, `originalCreator(tokenId)` MUST still return the original minter and `getOwnershipHistory(tokenId)` MUST return the full history. Implementations SHOULD use `originalCreator[tokenId]` (not `_currentOwner[tokenId]`) for existence checks in view functions so that Layer 2 queries survive burn.

#### ERC-165

16. Compliant contracts MUST implement [ERC-165](./eip-165.md) and return `true` for:
    - `0x01ffc9a7` ([ERC-165](./eip-165.md))
    - `0x80ac58cd` ([ERC-721](./eip-721.md))
    - The interface ID computed as the XOR of all function selectors in `IERC721H`
    - If `IERC721HAnalytics` is implemented, additionally the interface ID of `IERC721HAnalytics`

#### Analytics Extension (`IERC721HAnalytics`) — OPTIONAL

The following requirements apply only to contracts that implement the optional `IERC721HAnalytics` interface:

A1. `isEarlyAdopter(account, blockThreshold)` MUST execute in O(n) time where n is the number of tokens created by `account`. Implementations MUST NOT iterate over global token supply.
A2. `getEverOwnedTokensLength(account)` MUST return the length of the deduplicated per-address token list.
A3. `getEverOwnedTokensSlice(account, start, count)` MUST return a slice of at most `count` token IDs starting at index `start`.
A4. `getCreatedTokensLength(creator)` MUST return the length of the per-creator token list.
A5. `getCreatedTokensSlice(creator, start, count)` MUST return a slice of at most `count` token IDs starting at index `start`.
A6. `getOwnerAtTimestamp()` is DEPRECATED. It MUST be `pure` and MUST always return `address(0)`.
A7. All per-address pagination functions MUST return an empty array if `start >= length`. They MUST NOT revert.

## Rationale

### Why Three Layers Instead of One Extended Mapping?

A single `mapping(uint256 => OwnershipRecord)` struct was considered but rejected because:

- **Layer 1 and Layer 2 have different mutability guarantees.** Layer 1 is write-once; Layer 2 is append-only; Layer 3 is freely mutable. Mixing them in one struct obscures these invariants.
- **Separate mappings allow independent optimization.** `hasEverOwned()` uses a dedicated `mapping(uint256 => mapping(address => bool))` for O(1) lookups without touching the history array.
- **The mental model is clearer.** Developers can reason about "which layer am I modifying?" during code review.

### Why O(1) `hasEverOwned()` Instead of Array Scan?

The most common on-chain query is: *"Has this address ever owned this token?"* — used for governance, airdrops, and access control. A linear scan of the history array would make this O(n) where n is the number of transfers, which becomes unbounded over time. The dedicated `_hasOwnedToken` mapping guarantees constant-time lookups at a cost of one additional `SSTORE` per first-time owner per token.

### Why Preserve History After Burn?

Burning a token destroys its economic value but not its historical significance. A burned Bored Ape still has provenance value — who created it, who held it, when it was destroyed. Legal disputes, insurance claims, and historical research all benefit from preserved Layer 1 and Layer 2 data. The storage cost is already paid; clearing it would refund a small gas amount but destroy irreplaceable data.

### Why an Append-Only History Instead of a Merkle Tree?

Merkle trees compress historical data but require off-chain computation to verify membership. The entire point of this standard is that **other smart contracts** can query history natively. An append-only array is directly readable by any contract via `getOwnershipHistory()` without proof submission.

### Why Deduplicate `_everOwnedTokens`?

Without deduplication, a token bouncing between Alice and Bob (Alice → Bob → Alice → Bob) would add the token ID to each user's list on every transfer, growing unboundedly. The `_hasOwnedToken` mapping prevents this — each address appears in a token's per-address list at most once.

### Why Configurable History Modes?

Not all NFT use cases need complete on-chain history arrays. High-frequency gaming tokens may only need the O(1) `hasEverOwned()` check. Art provenance NFTs need the full trail. The three-mode system (FULL, FLAG_ONLY, COMPRESSED) widens adoption by letting deployers choose the storage/feature trade-off at construction time:

| Mode | Storage per transfer | getOwnerAtBlock() | getOwnershipHistory() | hasEverOwned() |
|:-----|:--------------------|:------------------|:---------------------|:---------------|
| FULL | ~4 SSTOREs | O(log n) ✓ | O(n) ✓ | O(1) ✓ |
| FLAG_ONLY | ~1 SSTORE | ✗ (returns 0) | ✗ (returns []) | O(1) ✓ |
| COMPRESSED | ~2 SSTOREs | ✗ (returns 0) | ✗ (returns []) | O(1) ✓ |

COMPRESSED mode provides an on-chain commitment (hash chain) that allows off-chain verification of the full history reconstructed from events, combining low gas cost with cryptographic provability.

### Why Split Core and Analytics?

The core interface (`IERC721H`) is intentionally minimal: origin tracking, O(1) membership checks, O(log n) historical resolution, and pagination. This covers the fundamental composability gap in ERC-721 — enabling other contracts to query ownership history natively.

Convenience functions like `isEarlyAdopter()`, `getProvenanceReport()`, and per-address token enumeration are useful but domain-specific. Moving them to an optional `IERC721HAnalytics` extension keeps the core interface small, orthogonal, and easier to implement. Contracts signal analytics support via `supportsInterface(type(IERC721HAnalytics).interfaceId)`. Minimal interfaces have a stronger track record of adoption across the ecosystem.

### Gas Trade-offs

Reference implementation measurements (Solidity 0.8.30, optimizer enabled, 200 runs) show the following approximate overhead relative to standard ERC-721:

| Operation | Approximate Overhead | Cause |
|:----------|:--------------------|:------|
| Mint | ~6–7× ERC-721 | 3 layers + Sybil guards + history arrays |
| Transfer | ~3–4× ERC-721 | 2 SSTOREs (history, block) + Sybil guards + dedup check |
| Burn | Comparable or lower | Layer 1 & 2 slots are not cleared; varies with storage warmth |
| Read | Free (view) | All queries are `view` |

Exact costs vary with compiler version, optimizer settings, and storage warmth. This overhead is most practical on Layer 2 rollups where write costs are significantly lower. On Layer 1, the higher cost is the explicit trade-off for permanent, trustless provenance.

## Backwards Compatibility

This standard is interface-compatible with [ERC-721](./eip-721.md), but implementations MAY introduce documented behavioral constraints (notably cooldown windows and self-transfer rejection). Wallets, marketplaces, and tooling that interact with ERC-721 can integrate without ABI changes, but execution assumptions that "approved + owned implies transfer always succeeds" may not hold when optional constraints are enabled.

The historical extension is additive at the storage/interface level:
- Existing ERC-721 contracts are **not** affected.
- No ERC-721 ABI changes are required.
- Contracts that implement this standard also implement ERC-721 by definition.
- `supportsInterface()` with the interface ID of `IERC721H` allows runtime detection.

### Migration Path for Existing Collections

Existing ERC-721 collections cannot be retroactively upgraded to comply with this standard (storage layout differs). The recommended migration pattern is:

1. Deploy a new contract implementing this standard.
2. Implement a `wrapAndMint(uint256 originalTokenId)` function that locks the ERC-721 token in the new contract and mints a wrapper token with the caller as `originalCreator`.
3. Historical ownership prior to wrapping is not captured — document this limitation to users.

## Test Cases

The reference test suite (current repository: 12 suites, 211 tests) validates the following critical behaviors:

- **Layer 1 invariants**: `originalCreator()` and `mintBlock()` remain immutable across transfers and burns.
- **Layer 2 append-only**: `getOwnershipHistory()` grows on every transfer and never shrinks. `hasEverOwned()` returns `true` for all past holders.
- **Layer 2 deduplication**: `getEverOwnedTokens()` contains each token at most once, even after round-trip transfers (A → B → A).
- **Layer 3 compatibility**: [ERC-721](./eip-721.md) functions (`ownerOf`, `transferFrom`, `safeTransferFrom`, `approve`, `setApprovalForAll`) retain ERC-721 interface/auth semantics; optional cooldown and self-transfer rejection are explicitly tested as documented divergences.
- **Burn preservation**: After burn, Layer 1 and Layer 2 remain queryable; Layer 3 (`ownerOf`) reverts.
- **Sybil guard (intra-TX)**: Same token transferred twice in one transaction reverts with `TokenAlreadyTransferredThisTx()`.
- **Sybil guard (inter-TX)**: Same token transferred twice in one block (separate transactions) reverts with `OwnerAlreadyRecordedForBlock()`.
- **Binary search**: `getOwnerAtBlock()` resolves the correct owner at arbitrary historical blocks, including blocks where no transfer occurred.
- **Pagination**: `getHistorySlice()`, `getEverOwnedTokensSlice()`, and `getCreatedTokensSlice()` return correct bounded slices and empty arrays for out-of-range inputs.
- **[ERC-165](./eip-165.md)**: `supportsInterface()` returns `true` for ERC-165, ERC-721, ERC-721 Metadata, and `IERC721H` interface IDs.

### Key Implementation Details

**Architecture (v2.1.x):**

The reference implementation uses a two-library architecture:

- **`ERC721HStorageLib`** — Low-level: HistoryStorage struct, `recordMint()`, `recordTransfer()`, binary search, Sybil guard, pagination.
- **`ERC721HCoreLib`** — High-level: `buildProvenanceReport()`, `getTransferCount()`, `isEarlyAdopter()`.

All library functions are `internal` — inlined at compile time for zero gas overhead.

A companion ERC-721HFactory.sol provides:
- **ERC-721HCollection** — Production wrapper: batch minting, supply cap, public mint with pricing/wallet limits, 5 batch historical query functions, configurable metadata URI.
- **ERC-721HFactory** — Permissionless CREATE2 deployer with deterministic cross-chain addresses and paginated registry.

**Storage Layout (HistoryStorage struct in ERC-721HStorageLib):**

```solidity
struct HistoryStorage {
    // Layer 1: Immutable Origin
    mapping(uint256 => address) originalCreator;
    mapping(uint256 => uint256) mintBlock;
    mapping(address => uint256[]) createdTokens;

    // Layer 2: Historical Trail
    mapping(uint256 => address[]) ownershipHistory;
    mapping(uint256 => uint256[]) ownershipTimestamps;
    mapping(uint256 => uint256[]) ownershipBlocks;        // O(log n) binary search support
    mapping(address => uint256[]) everOwnedTokens;
    mapping(uint256 => mapping(address => bool)) hasOwnedToken;
}

// Main contract:
using ERC721HStorageLib for ERC721HStorageLib.HistoryStorage;
using ERC721HCoreLib for ERC721HStorageLib.HistoryStorage;
ERC721HStorageLib.HistoryStorage private _history;
ERC721HStorageLib.HistoryMode private immutable _historyMode;
```

Note: The `_ownerAtBlock` mapping (used in versions prior to v1.5.0) has been eliminated. Inter-TX Sybil protection uses `lastTransferBlock[tokenId]`, which is updated in all history modes (FULL / FLAG_ONLY / COMPRESSED). The `getOwnerAtBlock()` query still uses O(log n) binary search over `_ownershipBlocks` when FULL mode arrays are present.

**Mint (virtual wrapper + internal primitive):**

```solidity
/// @notice Public mint — virtual so inheritors can add supply caps, allowlists, etc.
function mint(address to) public virtual onlyOwner returns (uint256) {
    return _mint(to);
}

/// @notice Internal mint — initializes all 3 layers. No access control.
/// @dev Callers MUST enforce authorization before calling.
///      Safe for inheriting contracts to build custom mint paths
///      (batch, public, allowlist) on top of this primitive.
function _mint(address to) internal returns (uint256) {
    if (to == address(0)) revert ZeroAddress();

    uint256 tokenId = _nextTokenId++;

    _beforeTokenTransfer(address(0), to, tokenId);

    // LAYER 1 & 2: Record origin + first history entry (via library)
    _history.recordMint(tokenId, to, block.number, block.timestamp, _historyMode);
    emit OriginalCreatorRecorded(tokenId, to);
    emit OwnershipHistoryRecorded(tokenId, to, block.timestamp);

    // LAYER 3: Set current owner (standard ERC-721)
    _currentOwner[tokenId] = to;
    _balances[to] += 1;
    _activeSupply += 1;

    emit Transfer(address(0), to, tokenId);

    _afterTokenTransfer(address(0), to, tokenId);

    return tokenId;
}
```

**Transfer (updates Layer 2 & 3, preserves Layer 1):**

```solidity
function _transfer(address from, address to, uint256 tokenId)
    internal nonReentrant oneTransferPerTokenPerTx(tokenId)
{
    if (_currentOwner[tokenId] != from) revert NotAuthorized();
    if (to == address(0)) revert ZeroAddress();
    if (from == to) revert InvalidRecipient(); // Prevent self-transfer (history pollution)

    _beforeTokenTransfer(from, to, tokenId);

    delete _tokenApprovals[tokenId];

    // SYBIL GUARD: One owner per token per block number (inter-TX)
    // Uses lastTransferBlock[tokenId], updated in all history modes.
    if (_history.isSameBlockTransfer(tokenId, block.number)) {
        revert OwnerAlreadyRecordedForBlock();
    }

    // LAYER 2: APPEND to ownership history (via library — auto-deduplicates)
    _history.recordTransfer(tokenId, to, block.number, block.timestamp, _historyMode);
    emit OwnershipHistoryRecorded(tokenId, to, block.timestamp);

    // LAYER 3: Update current owner (standard ERC-721)
    _currentOwner[tokenId] = to;
    _balances[from] -= 1;
    _balances[to] += 1;

    emit Transfer(from, to, tokenId);

    _afterTokenTransfer(from, to, tokenId);
}
```

**Burn (clears Layer 3, preserves Layer 1 & 2):**

```solidity
function burn(uint256 tokenId) public {
    address tokenOwner = ownerOf(tokenId);
    if (msg.sender != tokenOwner && !_isApprovedOrOwner(msg.sender, tokenId)) {
        revert NotApprovedOrOwner();
    }

    // Clear approvals
    delete _tokenApprovals[tokenId];

    // LAYER 1 & 2: PRESERVED (immutable history survives burn)

    // LAYER 3: Remove current owner
    _currentOwner[tokenId] = address(0);
    _balances[tokenOwner] -= 1;
    _activeSupply -= 1;

    emit Transfer(tokenOwner, address(0), tokenId);
    emit HistoricalTokenBurned(tokenId);
}
```

**O(1) History Check:**

```solidity
function hasEverOwned(uint256 tokenId, address account) public view returns (bool) {
    return _hasOwnedToken[tokenId][account];
}
```

## Security Considerations

### Storage Complexity Analysis

The following table formalizes the storage and compute complexity of each operation:

| Operation | Write Complexity | Storage Growth | Read Complexity |
|:----------|:----------------|:---------------|:----------------|
| `recordMint()` | O(1) — 5 SSTOREs (FULL) | +5 slots per mint | O(1) |
| `recordTransfer()` | O(1) — 3–5 SSTOREs (FULL) | +3 slots per transfer (new owner: +5) | O(1) |
| `hasEverOwned()` | — | — | O(1) — mapping lookup |
| `getOwnerAtBlock()` | — | — | O(log n) — binary search |
| `getOwnershipHistory()` | — | — | O(n) — full array return |
| `getHistorySlice()` | — | — | O(k) — bounded slice |
| `isEarlyAdopter()` | — | — | O(m) — m = tokens created by account |

**Per-transfer storage footprint (FULL mode):**

Each transfer appends to three parallel arrays and conditionally writes two more slots:

| Slot | Size | Frequency |
|:-----|:-----|:----------|
| `ownershipHistory[tokenId]` | 20 bytes (address) | Every transfer |
| `ownershipTimestamps[tokenId]` | 32 bytes (uint256) | Every transfer |
| `ownershipBlocks[tokenId]` | 32 bytes (uint256) | Every transfer |
| `lastTransferBlock[tokenId]` | 32 bytes (uint256) | Every transfer |
| `everOwnedTokens[owner]` | 32 bytes (uint256) | First time only |
| `hasOwnedToken[tokenId][owner]` | 1 byte (bool) | First time only |

**Worst-case storage per token (FULL mode):**

For a token with `T` transfers among `U` unique owners:
- Array slots: `3T` (history, timestamps, blocks)
- Per-owner slots: `2U` (everOwnedTokens entry + hasOwnedToken flag)
- Fixed slots: `3` (originalCreator, mintBlock, lastTransferBlock)
- **Total: `3T + 2U + 3` storage slots**

At 20,000 gas per cold SSTORE, a token with 1,000 transfers consumes ~3,003 slots ≈ 96 KB of state.

**FLAG_ONLY mode:** Storage per token = `U + 3` slots (flags + Layer 1). No arrays.

**COMPRESSED mode:** Storage per token = `U + 4` slots (flags + Layer 1 + hash). No arrays.

### Unbounded History Growth

The `_ownershipHistory` array grows by one entry per transfer. For heavily traded tokens (e.g., wash trading), this array could become very large. However:

- **Reading** the array via `getOwnershipHistory()` is a `view` call and costs no gas.
- **Writing** (appending) is O(1) and costs a fixed ~20,000 gas per SSTORE regardless of array length.
- **On-chain iteration** over the array (e.g., in `isEarlyAdopter`) should be avoided for tokens with large histories. Implementations MAY add a `maxHistory` cap.

### Reentrancy

The reference implementation uses a `nonReentrant` modifier on `_transfer()` to prevent reentrancy during the multi-step state update (Layer 2 append + Layer 3 update). Implementations MUST ensure that external calls (e.g., `onERC721Received`) occur **after** all state mutations.

### Privacy

All ownership history is permanently public on-chain. Users should be aware that transferring a token implementing this standard creates an **indelible, public record** linking their address to the token. This is by design (provenance requires transparency) but may conflict with privacy expectations. Users who require privacy SHOULD use fresh addresses for each interaction.

### Storage Cost Griefing

An attacker could repeatedly transfer a token to inflate `_ownershipHistory`. Each append costs the **sender** gas (not the contract), so the attacker pays for the attack. The `_everOwnedTokens` deduplication ensures that an address's per-address list grows at most once per unique token, limiting the blast radius.

### Pagination Anti-Griefing

All unbounded arrays (`_ownershipHistory`, `_everOwnedTokens`, `_createdTokens`) expose both full-return and paginated-slice functions. Frontends SHOULD prefer `getHistorySlice()`, `getEverOwnedTokensSlice()`, and `getCreatedTokensSlice()` for tokens or addresses with large histories to avoid RPC response limits or mobile wallet stalls.

### Binary Search Complexity

`getOwnerAtBlock()` uses O(log n) binary search over `_ownershipBlocks`. For a token with 1,000 transfers, this requires at most 10 comparisons — negligible gas for a `view` call. The `_ownershipBlocks` array is append-only and strictly monotonic (enforced by the Sybil guard), which guarantees binary search correctness.

### Transient Storage Key Namespacing

The intra-TX Sybil guard uses EIP-1153 transient storage. The transient storage key MUST be namespaced to prevent collision with other libraries or contracts executing in the same transaction context. The reference implementation uses `keccak256(abi.encode(tokenId, NAMESPACE_SALT))` where `NAMESPACE_SALT = keccak256("ERC721H.oneTransferPerTokenPerTx")`. Raw `tokenId` as a transient storage key is NOT safe — transient storage is global to the transaction, and other contracts or libraries may use overlapping key spaces.

### Adversarial Modeling: Churn Griefing Attack

**Attack vector:** An attacker transfers the same token once per block for `N` blocks to intentionally inflate the `_ownershipHistory` array, growing storage without economic constraint on L2 where gas is cheap.

**Cost model (FULL mode, L2):**

| Chain | Gas per transfer | ETH cost per transfer | Cost for 10,000 transfers |
|:------|:----------------|:---------------------|:--------------------------|
| Ethereum L1 | ~170,000 | ~$5–50 | $50,000–500,000 |
| Arbitrum | ~170,000 | ~$0.01–0.05 | $100–500 |
| Base | ~170,000 | ~$0.005–0.02 | $50–200 |

On L2, 10,000 transfers ≈ 30,000 additional storage slots ≈ 960 KB of permanent state — economically viable for an attacker at <$500 total cost.

**Mitigations (layered):**

1. **Same-block guard (built-in):** Prevents >1 transfer per token per block. Limits attack rate to 1 transfer per block time.
2. **Same-TX guard (built-in):** Prevents intra-TX chaining (A→B→C in one TX).
3. **Transfer cooldown (configurable):** `setTransferCooldown(blocks)` enforces a minimum gap between transfers. On L2, setting this to 300 blocks (~5 minutes on Arbitrum) limits an attacker to ~288 transfers/day, making the 10,000-transfer attack take ~35 days.
4. **FLAG_ONLY mode:** Eliminates array growth entirely. Storage overhead per transfer = 1 SSTORE (lastTransferBlock). Array-based griefing is impossible.
5. **COMPRESSED mode:** Eliminates arrays but preserves a hash-chain commitment. Storage overhead per transfer = 2 SSTOREs (lastTransferBlock + historyHash).

**Recommended deployment posture:**

| Context | Recommended Mode | Cooldown |
|:--------|:----------------|:---------|
| L1, low-turnover (art) | FULL | 0 (gas is natural deterrent) |
| L2, moderate turnover | FULL | 50–300 blocks |
| L2, high turnover (gaming) | FLAG_ONLY or COMPRESSED | 50+ blocks |
| Anti-wash-trading | FULL or COMPRESSED | 1800+ blocks |

### Formal Security Considerations

This section consolidates the principal risk surfaces and design tradeoffs reviewers should evaluate.

1. **State bloat (intentional tradeoff):** FULL mode grows persistent state linearly with transfers. This is not accidental; it is the core mechanism enabling on-chain provenance queries without indexers.
2. **Storage amplification risk:** Adversaries can increase Layer 2 footprint by repeatedly transferring a token. Mitigations are economic (attacker pays gas), architectural (FLAG_ONLY/COMPRESSED), and behavioral (same-block + cooldown constraints).
3. **Cooldown tradeoff:** Cooldown improves anti-grief posture but introduces a constrained ERC-721 behavior surface where approved transfers may temporarily revert.
4. **Event vs state duplication:** This standard duplicates ownership evidence from logs into state intentionally. Logs remain useful for indexing; state is required for contract-native composability.
5. **Compatibility assumptions:** Integrations that assume "approval + ownership implies immediate transfer success" MUST account for cooldown and self-transfer rejection.

### 10,000-Transfer Worst-Case Appendix

Let `T = 10,000` transfers for one token.

- **FULL mode storage growth (upper-bound):**
    - Array slots: `3T = 30,000`
    - Per-owner slots bounded by unique owners (`2U`), max `U <= T + 1`
    - Fixed slots: `3`
    - Approx upper-bound (worst unique-owner churn): `3T + 2(T+1) + 3 = 50,005` slots
- **FULL mode practical churn pattern (small owner set):**
    - Common wash pattern alternates among few addresses, so `U` stays small.
    - For small `U`, dominant term remains `3T`.

At 32 bytes/slot, 30,000 slots is ~960 KB of permanent state for the dominant array component alone.

### Cooldown Economics on L2

For cooldown `C` blocks and average block time `B` seconds:

- Max transfers/day per token = $\frac{86400}{C \cdot B}$
- Time to 10,000 transfers = $\frac{10000 \cdot C \cdot B}{86400}$ days

Example (Arbitrum-style, `B ≈ 1s`, `C = 300`):

- Max transfers/day ≈ 288
- 10,000-transfer grief campaign duration ≈ 34.7 days

This converts short-horizon griefing into a long-duration campaign, raising operational and detection costs.

### Differential Behavior Analysis (ERC-721 vs ERC-8169)

Reviewers should evaluate behavior under two regimes:

- **Compatibility regime (`transferCooldownBlocks = 0`):** ERC-721 interface and expected transfer flow remain familiar except intentional self-transfer rejection.
- **Defensive regime (`transferCooldownBlocks > 0`):** Transfers may revert within the configured window even when approved.

Marketplace and router integrations SHOULD:

1. Surface cooldown reverts to users as time-based constraints.
2. Retry only after block progression or explicit cooldown expiry checks.
3. Avoid optimistic assumptions that any approved transfer can execute immediately.

### Self-Transfer Divergence

This standard explicitly prohibits self-transfers (`from == to`), which diverges from canonical ERC-721 behavior where self-transfers are permitted. This prevents history pollution (a no-op transfer that adds a redundant entry). Indexers and marketplaces that assume self-transfer is valid SHOULD add explicit handling. The `InvalidRecipient()` revert makes this immediately detectable.

### Burn Does Not Delete

Burning preserves Layer 1 and Layer 2 storage. This means storage slots are **not** refunded on burn, unlike standard ERC-721. This is intentional — the historical data has value — but implementers should document this behavior.
